# Chapter 5 — Book Scaffold: `new_book.py`

*Thirty seconds of work makes the structural commitment that the rest of the book will either keep or betray.*

---

You open your book folder in Cowork. Your BLUEPRINT.md lives there. You tell Cowork what you want — build the scaffold from this blueprint under the slug `ai-for-designers` — and Cowork runs the command for you:

```bash
python new_book.py --blueprint BLUEPRINT.md --slug ai-for-designers
```

Thirty seconds later Cowork has finished. Forty lines of green checkmarks scroll past in its activity log. The last line says `done`. You look in the folder. There is a directory named `ai-for-designers/` that did not exist a minute ago. Inside it, files you did not write — `vision.md`, `architecture.md`, `chapters-spec.md`, `risks.md`, `outline.md`, `book.md`, `metadata.yaml`, `build.sh`, `README.md` — and folders: `chapters/`, `pantry/`, `images/`, `d3/`, `output/`, `styles/`.

Now notice what you feel.

A designer with ten years of client work just had a folder generated by a script Cowork ran. The instinct is to back away. The instinct says: *this is someone else's domain, this is complications I can't see, this is a door I shouldn't open*. The instinct is wrong, but it is real, and it is the chapter's only real obstacle. Most authors who quit at this point quit because they cannot read the directory — they see strangeness and assume complication. The chapter's only job is to collapse that strangeness into recognition. Everything the directory contains is something you already understand; it just has a filename instead of a name you know.

Start at the top of the directory. There are three audiences in there, and once you can see the three audiences, the rest reads itself.

---

Before the audiences, the command itself. You never type it into a terminal — you tell Cowork what you want and Cowork runs it. But it is worth understanding what Cowork is running. Two arguments. That is the entire interface.

`--blueprint` points to the BLUEPRINT.md file from Chapter 4. The script reads it. It looks for the chapter list, the metadata block, the three-act arc, and the capability statements. These become the scaffold's structural commitments — the empty chapter files waiting to be drafted, the stub specs, the named gaps. If the BLUEPRINT.md is missing a chapter title or a capability statement, the scaffold reveals it. Empty files, blank specs, named gaps. This is a feature, not a bug. The first time the BLUEPRINT.md is held to a deterministic standard is the first time its vagueness becomes visible. The scaffold is a test the spec either passes or fails.

`--slug` is the directory name. It becomes the project folder, the EPUB filename, and the URL slug if the book ever goes online. Kebab-case, lowercase, no spaces: `ai-for-designers`, not `AI for Designers`. The slug is the only argument you cannot easily change later — every other field in the scaffold is editable; the slug is baked into paths. Choose it once and choose it as if it will live on a shelf.

No interactive prompts, no global config file, no choices to make beyond those two. This is by design. Audrey Roy Greenfeld's Cookiecutter established the canonical Python scaffolding interface: one command, sensible defaults, no global state.[^cookiecutter] If you find yourself wanting more flags, you are usually about to make a decision you should have made in BLUEPRINT.md instead.

One practical note on running the command twice: the script refuses to overwrite. If Cowork runs it again in the same directory, nothing is destroyed. If you genuinely want to rebuild from scratch, tell Cowork to delete the directory first. The script's full usage reference — flags and output structure — is in Appendix C. If Cowork reports that running `new_book.py` printed `command not found`, Python needs to be installed first. You do not open a terminal to do this — you tell Cowork to install Python, and Cowork runs the right steps for your machine. The commands below are what Cowork runs behind the scenes. The rest of the chapter assumes Python 3.10 or newer.

**Mac.** Cowork installs Homebrew and Python for you:

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
brew install python
python3 --version
```

The last line should print `Python 3.12.x` or similar. If `brew` is unknown after the first step, Cowork may need to refresh its environment — Homebrew sometimes needs a PATH refresh.

**Windows.** Cowork installs Python via winget:

```powershell
winget install --id Python.Python.3.12 -e
python --version
py -m pip --version
```

The second line should print `Python 3.12.x`. If `winget` is missing, install App Installer from the Microsoft Store first.

**Linux (Ubuntu/Debian).** Cowork installs Python via apt:

```bash
sudo apt update && sudo apt install -y python3 python3-pip
python3 --version
pip3 --version
```

Both lines should print without error.

No virtual environments, no `pyenv`, no version managers. Five minutes, and Cowork does the typing. Philip Guo's 2014 documentation of Python's rise as the teaching language at top U.S. universities included one persistent finding: install friction was the single largest predictor of disengagement in the first week.[^guo] The install is handled here, by Cowork, because install friction is a real obstacle and a real insult to a person who has been writing professional briefs for a decade. You are not learning Python in this book. Cowork is running one script that someone else wrote so that a directory appears.

[^guo]: Guo, P. (2014). "Python is Now the Most Popular Introductory Teaching Language at Top U.S. Universities." *Communications of the ACM* blog. cacm.acm.org.

Back to the three audiences. This is the taxonomy the directory makes visible, and it is the frame that makes everything else in the scaffold comprehensible.

**Cowork reads some files at runtime.** When Cowork generates a chapter draft, it reads a specific set of files in the directory and ignores everything else. These are the machine-facing files — the spec, the research, the cross-chapter summary. If you change something in one of these files, the next draft reflects the change. If you change something outside these files, the next draft does not notice.

**You read some files while working.** The human-facing files are the working notebook. They were generated from your BLUEPRINT.md, but they are not read by the automation. They are the files where you think, where you orient, where you check whether a new chapter idea belongs in this book at all.

**Pandoc reads some files at build time.** The build-facing files feed the EPUB and PDF pipeline. They are not read during drafting; they become relevant only in Chapter 12 when the manuscript becomes an artifact.

---

The Cowork-facing files deserve a closer look because they are the files where a mistake propagates into every chapter that follows.

`BLUEPRINT.md` is the contract. The Chapter Writer prompt reads it first. The voice section, chapter list, capability statements, and bridge questions are all consumed at runtime. If a single file in the directory is the book's binding commitment, this is it. When you reach Chapter 8 and start rewriting Cowork drafts, the question to ask of any draft that seems off is not *what did Cowork do wrong?* but *what in BLUEPRINT.md is ambiguous?* The answer is almost always in BLUEPRINT.md.

`book.md` is the cross-chapter coherence document. Cowork uses it to know what's coming later in the manuscript while drafting an early chapter. It is generated from BLUEPRINT.md but may be edited. The only time you need to edit it is when the book's arc changes substantially — not when a chapter gets longer or shorter, but when a whole act shifts.

`pantry/` is the research layer, one notes file per chapter. Cowork reads the pantry file for the chapter it is currently drafting. It cannot look at any other chapter's pantry file while it works. This is intentional — cross-chapter contamination in research produces drafts that feel unfocused, as if the author is introducing things the reader hasn't seen yet. The pantry is Chapter 6's subject; for now, know only that it is the file Cowork reads when it needs domain knowledge that is not in the spec.

`chapters/` is where Cowork writes and where the build script reads. The numbering prefix is not decoration: `01-`, `02-`, `03-` is the concatenation order that becomes the EPUB's chapter order. Change the number, change the position in the book.

The file-level separation between Cowork-facing and human-facing files is older than AI tooling. Donald Knuth named the principle in 1984 as literate programming: "instead of imagining that our main task is to instruct a computer what to do, let us concentrate rather on explaining to human beings what we want a computer to do."[^knuth] The directory splits the explaining (human files) from the instructing (machine files) at the filesystem level. This is not a clever trick. It is a discipline, and it matters precisely because the temptation to blur the line is constant.

[^knuth]: Knuth, D. E. (1984). "Literate Programming." *The Computer Journal*, 27(2), 97–111.

---

The human-facing files are the ones that keep you from losing the plot.

`vision.md` is the one-paragraph statement of what the book is and who it is for. Generated from the BLUEPRINT.md vision section. Read it when you have drifted. Update it when the book's purpose actually changes — not when a chapter is harder than expected, not when a section runs long, but when the fundamental answer to *who is this for and what does it help them do* has genuinely shifted. That happens rarely.

`architecture.md` is the three-act arc and the prerequisite map. When you are deciding whether a new chapter idea belongs in this book at all, this is the file to open. A chapter that does not fit the arc is not a bad chapter — it may be the seed of a different book. `architecture.md` is how you tell the difference.

`chapters-spec.md` is the full chapter list with capability statements, learning outcomes, and bridge questions. This is your editor's reference. In Chapter 8, when you are rewriting Cowork drafts, the question *does this draft do what the chapter was supposed to do?* is answered by comparing the draft against the spec in this file.

`risks.md` is the contested claims register. It is generated from BLUEPRINT.md Part 11 and is meant to be read before you publish, not before you draft. The aging risk register — claims that depend on current labor-market data, model behavior, tool availability — lives here. It is the file that reminds you which claims need to be verified again before the book goes live.

`outline.md` is the human-readable table of contents. Copy it into a Substack newsletter, paste it into an email to a colleague, show it to a potential student. It is the book's public face before the book exists.

There is one rule about all of these files: do not let Cowork edit them. The temptation, when Cowork produces a strong draft, is to ask it to update `architecture.md` to reflect the new direction. Resist. These are the files where you think. Letting the automation rewrite them is letting the automation think about what the book is. That is the fluency trap one floor up from where Chapter 1 located it.

---

The build-facing files are the simplest to understand, and the most useful to understand early even though you will not use them until Chapter 12.

`metadata.yaml` is what pandoc reads at build time. Eleven fields. Every one of them lands in the EPUB metadata. Here is the fully populated version for `ai-for-designers`:

```yaml
---
title: "AI for Designers"
subtitle: "A Practitioner's Guide to AI+1 for Graphic Design"
author: "Your Name"
publisher: "Your Studio Press"
rights: "© 2026 Your Name. All rights reserved."
language: en-US
date: 2026-05-28
description: >
  A short practitioner handbook for working graphic designers
  building one client-facing or workshop-ready AI+1 textbook.
identifier: ai-for-designers-v1
cover-image: cover.jpg
toc: true
toc-depth: 2
---
```

Populate every field now, with real values, not placeholders. The values you put here will appear in the EPUB Kindle generates. Greg Wilson's Software Carpentry work found that people read directories and metadata before they read code — populated artifacts signal that someone was actually here, making real decisions.[^wilson] An eleven-field `metadata.yaml` full of `<your-title-here>` placeholders is not a scaffold; it is a gesture toward a scaffold. Fill it in.

`build.sh` is three things doing all the work:

```bash
#!/usr/bin/env bash
cat chapters/*.md > _working/combined.md
pandoc _working/combined.md \
  --metadata-file=metadata.yaml \
  --epub-cover-image=cover.jpg \
  -o output/${SLUG}.epub
pandoc _working/combined.md \
  --metadata-file=metadata.yaml \
  --pdf-engine=xelatex \
  -o output/${SLUG}.pdf
```

Concatenation: every file in `chapters/` is glued together in filename order. This is why the numbering prefix matters. The number is the build order. Pandoc: the universal document converter takes Markdown plus a metadata file and produces EPUB and PDF.[^pandoc] The `metadata.yaml` schema is pandoc's own, not invented by this pipeline. Output: everything lands in `output/`, which is gitignored. The output directory is not the book. The book is in `chapters/`. Treat `output/` as disposable — if you delete it, the next time Cowork runs the build it recreates it in under a minute.

The Turing Way community calls a build script "the covenant that anyone can reproduce the artifact."[^turingway] `build.sh` is your covenant with future-you. Six months from now, after a typo fix, you tell Cowork to run the build and it does. No memory required. Quarto and Jupyter Book offer more elaborate build systems for scientific publishing; they are excellent and they are overkill for a practitioner handbook produced by one author. `new_book.py` ships pandoc and bash because that combination has been stable for fifteen years and will be stable for another ten.

What the scaffold actually establishes — beneath the filenames and the three audiences — is that the BLUEPRINT.md session was the real work.

This is not obvious from inside Chapter 4. Inside Chapter 4, the Blueprint session feels like preparation for writing. The scaffold makes visible what was actually true: the session was the structural commitment. Every chapter file in `chapters/` is named from a decision made in that session. Every capability statement in `chapters-spec.md` is a promise made in that session. Every contested claim in `risks.md` is a flag raised in that session. The directory is not the output of the script; it is the output of the conversation with Blueprint, made legible as a filesystem.

Kristen Nygaard, who invented object-oriented programming with Ole-Johan Dahl by building Simula in the 1960s, held a central conviction that programs are models of the world — that the structure of the program must mirror the structure of what is being modeled.[^nygaard] The scaffold is Nygaard's idea applied to a book. Your BLUEPRINT.md describes a book. `new_book.py` produces a directory that mirrors that description. Chapters are files. The audience structure mirrors the production structure. If the model is wrong — if the BLUEPRINT.md was vague, or if the chapter list did not survive the structural pressure of the intake questions — the directory is wrong, and the wrongness shows up at every step that follows. The scaffold does not fix a bad spec. It reveals one.

This is why Chapter 4 is the hardest chapter in the book, and why this chapter is comparatively short. The difficulty was front-loaded. The scaffold rewards that front-loading by converting a Word document into a directory where real work can begin.

[^nygaard]: Nygaard, K., & Dahl, O.-J. (1978). "The Development of the SIMULA Languages." *ACM SIGPLAN Notices*, 13(8), 245–272.

## Bridge — Chapter 6

The directory exists. `BLUEPRINT.md` is in place. `chapters/` is full of empty stubs. But Cowork cannot draft yet. It does not know what sources to cite, which domain examples land, which contested claims to flag. It has the spec but not the research.

The pantry is empty.

Chapter 6 runs the Chapter Research Gatherer — the automation that produces one notes file per chapter — and then teaches you to read those notes critically before handing them to the drafting system. The pantry is what stands between a Cowork draft worth rewriting and a Cowork dump. The difference, in practice, is whether someone did the Chapter 6 work first.
